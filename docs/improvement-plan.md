# llm-council Improvement Plan

_Items 1–7 are new, user-facing quality items grounded in the research notes under `azalio-obsidian/azalio/sources/`. They target what a user actually feels when calling `ask_council`: trust, calibrated uncertainty, clarification, latency, and source attribution. Items 8–11 are user-facing slices retained from the prior plan. Engineering and research-flavoured items with no direct user signal are consolidated under "Deferred" at the bottom for traceability._

## Add a semantic answer cache so users don't pay full council cost for repeat or near-duplicate questions [2605.007]

**Status (2026-05-18)**: Completed as an umbrella; do not execute directly. Slice `2605.007-1` shipped a conservative first-turn repeat-answer cache for context-free default-auto requests. Slice `2605.007-2` shipped local semantic near-duplicate matching plus chairman validation for borderline matches. Slice `2605.007-3` shipped runtime cache KPIs and an offline replay probe so operators can verify hit latency, match types, and validation outcomes before changing thresholds.

**Source**: [[continuous-semantic-caching-for-low-cost-llm-serving]] (paper note)
**Implementation Layer**: `backend/answer_cache.py`, `backend/storage.py`, `mcp_server/server.py:ask_council`, and `backend/main.py` before model fan-out
**Missing Capability**: A semantic cache that recognises when a new question is sufficiently similar to a previously answered one and serves the prior chairman synthesis (with an explicit "cached" marker) instead of re-running the full council.
**Architecture Evidence**: The MCP tool currently runs the full 3-stage pipeline for every call, even when the same user types `summarise the previous discussion` twice in a row, or when two team members ask the same question minutes apart. `backend/storage.py` already persists every chairman synthesis in SQLite, so the corpus to cache against already exists; only the embedding index and the lookup path are missing.
**Benefit Hypothesis**: When a semantic cache layer is added with an ε-neighbourhood policy, then on a real-traffic probe (replayed conversation log) ≥20% of MCP calls served from cache, end-to-end median latency on cache hits drops to ≤500 ms, and answer quality on cache hits stays within 5% of a freshly-deliberated baseline (measured by the LLM-as-a-judge from item [2604.011]).
**Confidence**: 0.58
**Reasoning**: The Continuous Semantic Caching paper provides both the theoretical framing (ε-net over Voronoi regions, sublinear regret) and the empirical signal (43–73% suboptimality reduction). The council's repeat-question pattern is exactly the workload it was designed for. Risks: (a) staleness of cached answers when the underlying question is time-sensitive; (b) cache hits on superficially similar but materially different questions. Both are addressable with an explicit time-to-live and a small `chairman: confirm cached answer still applies` re-validation pass on borderline matches.
**Why Not Already Tried**: The plan currently has no caching item; prior items focus on cost via routing or augmentation rather than via re-use. Caching also requires an embedding pipeline, which was out of scope for the early build.

## Cascaded partial-evidence ranking for Stage 2 council selection [2605.008]

**Source**: [[caps-cascaded-adaptive-pairwise-selection-for-efficient-parallel-reasoning]] (paper note)
**Implementation Layer**: `backend/council.py` Stage 2 ranking / thorough-mode candidate selection, `run_status` metadata, `backend/metrics.py`, and mocked-provider tests.
**Missing Capability**: Stage 2 currently ranks full council responses as a single expensive step; it has no cheap evidence levels, early elimination, or finalist-only comparison path for larger/deep council runs.
**Architecture Evidence**: `docs/architecture.md` defines the core runtime as Stage 1 parallel model queries, Stage 2 anonymous ranking, and Stage 3 chairman synthesis. It also documents async fan-out, configurable timeouts, saved conversation metadata, and MCP progress/heartbeat for long runs, so the project owns the exact orchestration and observability layer where a ranking-cost cascade would live.
**Benefit Hypothesis**: On a benchmark with at least five candidate answers or deep/thorough revision candidates, a cascaded ranking mode will reduce Stage 2 ranking tokens by at least 35% and Stage 2 wall-clock latency by at least 20% while preserving the full-ranking top answer within 5 percentage points. For the current small council size, keep the mode disabled unless the same benchmark shows net savings after the extra control logic.
**Confidence**: 0.52
**Reasoning**: CAPS is useful here after mechanism translation: its domain is code/math self-verification, but the portable mechanism is a budgeted selection cascade that starts from cheap candidate views and spends full-evidence judge calls only on finalists. That maps directly to llm-council's Stage 2 answer-selection cost, especially when `deep` mode or future expanded councils create more candidate responses. The fit is bounded because today's default council is small; this should be a measured opt-in path, not a replacement for the existing complete anonymous ranking.
**Why Not Already Tried**: Existing plan items use Stage 2 agreement after complete rankings for confidence, escalation, and early stopping. None changes how Stage 2 obtains the ranking itself, and none defines partial-answer signatures, low-evidence eliminations, or finalist-only comparisons.
**Pairs With**: [2605.006] complexity-aware mode selection, [2604.093] confidence-aware selective fallback, and "Compute-aware early stopping for thorough council runs".

### Proposed Changes

- Add a Stage 2 ranking strategy enum: `full` (current default) and `cascade` (opt-in for benchmarked large/deep candidate pools).
- Build an evidence map for each candidate answer: stable digest/outline, key claims only, and full response. The digest/outline level should be deterministic and stored in `run_status` for audit.
- In cascade mode, eliminate obvious duplicates or weak candidates using cheap views first, then run full-response comparisons only among finalists; include a rescue pass when cheap evidence conflicts with full-ranking confidence.
- Preserve the existing complete-ranking metadata shape by recording both the cascade winner and enough audit fields to explain which candidates were eliminated at which evidence level.
- Add mocked-provider tests that compare cascade vs full ranking over fixed candidate sets, plus a benchmark script that reports ranking tokens, latency, top-1 agreement, and low-confidence false-pass rate before enabling the strategy by default.


## Position-order debiasing for Stage 2 ranking and the judge via dual-order evaluation + majority vote [2606.03189a]

**Source**: [[sensejudge-human-centric-preference-driven-judgment]] (paper note) — arXiv:2606.03189, "SenseJudge: Human-Centric Preference-Driven Judgment Framework" (Li, Liu, Kong, Xu, Sui; PKU / StepFun / XJTU), https://arxiv.org/abs/2606.03189, code/data https://github.com/qiongrenpiqida/SenseJudge
**Implementation Layer**: `backend/council.py` (`stage2_collect_rankings`, `calculate_aggregate_rankings`, `compute_council_confidence`), `backend/eval/judge.py`, with mocked-provider coverage in `tests/test_council_confidence.py` and `tests/test_judge_eval.py`.
**Missing Capability**: Stage 2 anonymises responses as fixed-order "Response A, B, C…" and asks each council member to rank them in that single presentation order; the judge likewise scores a single (candidate, baseline) presentation. Neither path measures or neutralises label/position bias — the well-documented tendency of LLM judges to favour whichever answer appears first (or under a particular letter).
**Architecture Evidence**: `docs/architecture.md` and `CLAUDE.md` describe Stage 2 as anonymous ranking over `label_to_model` mappings, feeding `calculate_aggregate_rankings` and `compute_council_confidence` (top-1 stability, Kendall-like agreement, disagreement score). The confidence signal is already consumed downstream (chairman hedge instruction, auto-standard escalation, adaptive routing expansion), so any order-induced bias in rankings silently propagates into escalation and routing decisions. `backend/eval/judge.py` already selects a single best Stage 1 baseline and runs one judge call per synthesis.
**Benefit Hypothesis**: When each pairwise/ranking judgment is evaluated in both presentation orders and reconciled by majority vote (ties → "inconsistent"), then on a mocked fixed-candidate benchmark the order-flip disagreement rate is measurable and reported as a new `compute_council_confidence` field, and the share of rankings that flip top-1 purely on label order drops to near zero, with chairman top-1 agreement preserved within 5 percentage points of the current single-order path. For the judge, dual-order scoring removes position-driven wins on `judge_eval_smoke` without changing the `judge.v1` schema contract.
**Confidence**: 0.6
**Reasoning**: SenseJudge makes dual-order evaluation a load-bearing part of its method (it explicitly scores `(r1,r2)` and `(r2,r1)` and uses a strict match), and its measured +16.08% average gain over baselines on SenseBench is partly attributed to that debiasing plus preference aggregation. DGAO (arXiv:2605.11974) independently corroborates the problem's severity — even frontier models (Claude-Sonnet-4, GPT-4o) stay below a 100% Consistency Rate across 8 random orders and degrade further as context grows — and contributes a critical caveat: naively *forcing* order-consistency produces "overconfident" consistent-but-wrong outputs (its Overconfidence Rate metric), so order-agreement must never be rewarded on its own, only jointly with correctness/quality. The mechanism is provider-agnostic and cheap relative to council fan-out (≤2× the Stage 2 / judge call count, and the judge is operator-facing, off the `ask_council` hot path). The bounded risk is added latency and the need to define a reconciliation rule for order-disagreements; surfacing disagreement as a confidence signal is strictly more information than today.
**Why Not Already Tried**: Existing items tune Stage 2 *aggregation* (cascade ranking [2605.008]), *when* extra stages run (early stopping, escalation), and confidence *consumption*, but none touches how a single ranking/judgment is *elicited*. Position/label bias has never been measured in this repo; the confidence signal currently assumes each ranking is an unbiased read.
**Pairs With**: [2605.008] cascaded Stage 2 selection (both restructure how rankings are produced), [2604.011] LLM-as-a-judge harness (the judge inherits the same debiasing), and [2605.11974] DGAO (its Consistency-Rate / Overconfidence-Rate metrics are the measurement framework; its RL fine-tuning method itself does NOT apply to our frozen-API council).

### Proposed Changes

- Add a `dual_order` evaluation mode to Stage 2 ranking: present the anonymised set in the canonical order and in a deterministic reversed/rotated order, then reconcile per-evaluator rankings by majority vote, marking irreconcilable pairs as order-inconsistent.
- Extend `compute_council_confidence` with an `order_consistency` (a.k.a. position-bias) field derived from how often a model's pairwise preferences flip under reordering; expose it in saved metadata alongside the existing top-1 stability and disagreement scores.
- Apply the same both-order trick to `backend/eval/judge.py`: score (candidate, baseline) in both presentation orders and require agreement before recording a winner; on disagreement, mark the judge result `available=true` but `winner="inconsistent"` rather than letting position decide.
- Keep `full` single-order as the default; gate `dual_order` behind config so the extra call cost is opt-in, and report order-flip rate in the benchmark before enabling by default.
- Per DGAO's Overconfidence-Rate caveat, never treat order-consistency as quality on its own: a response that ranks identically under all orders may be consistently *wrong*. Cross the `order_consistency` signal with the judge/aggregate-quality signal before using it to raise confidence, and in the benchmark report consistency split by correct vs incorrect (robust-consistency vs overconfident-consistency).
- Add mocked-provider tests covering: stable-across-orders, flips-on-order, and tie cases, for both Stage 2 and the judge; update `scripts/judge_eval_smoke.py` to emit the per-order agreement rate.


## Preference-set ensemble for the LLM-as-a-judge pipeline [2606.03189b]

**Source**: [[sensejudge-human-centric-preference-driven-judgment]] (paper note) — arXiv:2606.03189, https://arxiv.org/abs/2606.03189
**Implementation Layer**: `backend/eval/judge.py` (`DEFAULT_JUDGE_RUBRIC`, `judge.v1` schema, judge model invocation), config knobs in `backend/config.py` (`JUDGE_*`), `tests/test_judge_eval.py`, and `scripts/judge_eval_smoke.py`.
**Missing Capability**: The judge currently scores a synthesis against the best Stage 1 baseline with a *single* fixed rubric (`DEFAULT_JUDGE_RUBRIC`) in one model call. There is no notion of multiple evaluation perspectives, no selection of which perspective generalises best, and no ensemble vote — so a single brittle rubric phrasing or a single model sample fully determines the operator-facing score.
**Architecture Evidence**: `CLAUDE.md` documents `backend.eval.judge` as a post-hoc, operator-facing measurement that returns `available=false` on failure and validates every rubric score under the strict `judge.v1` schema. The judge is explicitly *not* part of the `ask_council` answer contract and runs off the request hot path, which makes it the safe place to spend extra model calls for robustness.
**Benefit Hypothesis**: When the judge runs N rubric/preference variants and aggregates them by instance-level majority vote (keeping the existing `judge.v1` per-score validation per variant), then on a fixed labelled probe the judge's agreement with held-out human/reference preferences improves over the single-rubric baseline, and run-to-run variance of the verdict drops, with `available=false` still returned only when *all* variants fail. No change to the public schema beyond an optional `ensemble` metadata block.
**Confidence**: 0.5
**Reasoning**: SenseJudge's core result is that distilling several explicit textual preferences, selecting the best-performing subset, and majority-voting beats both prompt-only frontier judges (GPT-4o 63.98 overall) and trained reward models (Skywork-Reward-Gemma2-27B 64.70), reaching e.g. 82.65 for Qwen3-14B + SenseJudge on their SenseBench. The portable mechanism — *ensemble of evaluation perspectives + selection + majority vote* — maps cleanly onto our single-rubric judge. The fit is bounded: the paper's "preferences" are distilled from labelled `(query, chosen, rejected)` data we do not collect, so for now the ensemble members are rubric variants (concision-weighted, correctness-weighted, evidence-weighted, etc.) rather than per-user-distilled preferences, and the +16% figure is on the authors' own benchmark, not ours.
**Why Not Already Tried**: The judge was added as a single-rubric measurement artifact ([2604.011] / commit `70c8d62`); the plan has no item about judge *robustness* or multi-perspective aggregation, only about *using* judge scores as a quality gate for caching and routing.
**Pairs With**: [2606.03189a] dual-order debiasing (each ensemble member also runs both orders) and [2604.011] judge harness.

### Proposed Changes

- Define a small set of named rubric/preference variants alongside `DEFAULT_JUDGE_RUBRIC` (e.g. correctness-first, concision-first, evidence/attribution-first), each validated independently under `judge.v1`.
- Run the variants over the same (candidate, baseline) pair and aggregate winners by majority vote at the instance level; record per-variant verdicts in an optional `ensemble` metadata block while keeping the top-level contract (`available`, `winner`, `scores`) unchanged.
- Add a labelled offline probe (candidate SenseBench, license permitting — verify the GitHub repo's terms before vendoring any data) to `scripts/judge_eval_smoke.py` to measure ensemble vs single-rubric agreement and verdict variance before enabling the ensemble by default.
- Keep single-rubric as the default `available=false`-on-all-fail behaviour; gate the ensemble behind a `JUDGE_ENSEMBLE_*` config so extra judge-model cost is opt-in.
- Extend `tests/test_judge_eval.py` with unanimous, split-majority, and all-fail cases to lock the aggregation and failure semantics.


## Reliability- and correlation-weighted Stage 2 aggregation instead of equal-weight average rank [2510.01499]

**Source**: [[beyond-majority-voting-llm-aggregation-by-leveraging-higher-order-information]] (paper note) — arXiv:2510.01499, "Beyond Majority Voting: LLM Aggregation by Leveraging Higher-Order Information" (Ai, Pan, Simchi-Levi, Tambe, Xu; MIT / Harvard / UChicago), https://arxiv.org/pdf/2510.01499
**Implementation Layer**: `backend/council.py` (`calculate_aggregate_rankings`, `compute_council_confidence`, Stage 2 top-1 selection feeding Stage 3), an offline evaluator-reliability estimator over stored conversations in `backend/storage.py`, and mocked-provider tests in `tests/test_council_confidence.py`.
**Missing Capability**: `calculate_aggregate_rankings` treats every council member's ranking with equal weight (plain average rank position) — a zero-order, majority-voting-style aggregate. It has no per-evaluator reliability weighting and no mechanism to discount evaluators whose errors are correlated (e.g., two same-family models that mis-rank together), so a correlated bloc can dominate the aggregate even when an independent minority is right.
**Architecture Evidence**: `CLAUDE.md` documents `calculate_aggregate_rankings` as "average rank position across all peer evaluations" and `compute_council_confidence` as the only higher-order read (top-1 stability, Kendall-like agreement, disagreement). The repo already treats evaluator heterogeneity as load-bearing — config startup *fails* if the chairman shares a provider family with the council — which is exactly the correlated-error structure this paper formalises. `backend/storage.py` persists every past ranking, so a corpus to estimate evaluator reliability/correlation already exists.
**Benefit Hypothesis**: When Stage 2 top-1 selection uses the paper's Optimal-Weight (first-order, inverse-logit of per-evaluator accuracy) and/or Inverse-Surprising-Popularity (second-order, label-free correlation-aware) aggregation instead of equal-weight average rank, then on a labelled offline probe of contested questions the aggregate top-1 agrees with the reference-best answer more often than equal-weight averaging, with the largest gains on low-`compute_council_confidence` (contested) cases and negligible change on unanimous ones — matching the paper's finding that gains are modest on easy items but >7% on hard/strong-distractor subsets (MMLU-hard: OW 24.79% vs MV 17.23%).
**Confidence**: 0.5
**Reasoning**: The paper proves Optimal Weight is the Bayesian-optimal aggregator (and reduces to majority voting only when evaluators are equally accurate — Corollary 3.3), and that the label-free ISP estimator beats majority voting in expectation (ISP ≥ MV ≥ SP), validated on UltraFeedback/MMLU/ARMMAN with significant t-stats (12.53/23.39/3.22, p<0.001). The portable mechanism — *weight evaluators by estimated reliability and down-weight correlated ones, all without labels* — maps directly onto Stage 2 and directly attacks the same-family correlated-error risk the repo already guards against structurally. Bounded fit: (a) it assumes a discrete K-way label space, so it applies cleanly to top-1 *selection* but not to full ordinal rankings without adaptation; (b) second-order estimation needs many questions over a *stable* evaluator set, so it must be windowed and invalidated when `COUNCIL_MODELS` changes; (c) absolute gains on easy questions are small, so this should fire only on contested runs.
**Why Not Already Tried**: Every existing aggregation item ([2605.008] cascade, confidence escalation, early stopping) changes *how much* ranking work runs or *how* confidence is consumed, but all consume a flat equal-weight aggregate. No item weights evaluators by reliability or models inter-evaluator correlation.
**Supporting Theory (folded in)**: "Multi-Agent Systems are Mixtures of Experts" (arXiv:2605.25929) models LLM deliberation with Friedkin–Johnsen opinion dynamics and shows the final answer is already a convex combination of members' positions with *input-dependent* weights — i.e. deliberation behaves like a confidence-weighted mixture-of-experts whose influence routing is driven mainly by relative confidence and initial alignment. That makes the equal-weight aggregate the council uses an implicit, uncontrolled version of what OW/ISP make explicit and reliability-grounded — supporting this item's premise that weights should be principled rather than uniform.
**Pairs With**: [2606.03189a] dual-order debiasing (both clean up Stage 2 selection quality), [2605.008] cascade ranking, and the chairman-heterogeneity invariant in `tests/test_chairman_heterogeneity.py`.

### Proposed Changes

- Add an offline evaluator-reliability job: over a rolling window of stored conversations with a stable `COUNCIL_MODELS` set, estimate each model's ranking accuracy and pairwise agreement/correlation from the persisted Stage 2 data (label-free, second-order as in the paper's empirical conditional-probability estimator).
- Introduce an aggregation-strategy enum in `calculate_aggregate_rankings`: `equal` (current default), `optimal_weight` (inverse-logit reliability weights), and `isp` (correlation-aware, label-free); apply the weighted aggregate to top-1 selection while keeping the displayed per-model average ranks unchanged for transparency.
- Gate the weighted strategy to fire only when `compute_council_confidence` reports low confidence (contested run), where the paper predicts the gain concentrates; on unanimous runs keep equal-weight.
- Invalidate/retrain reliability estimates whenever `COUNCIL_MODELS` changes (tie into the existing chairman-heterogeneity startup check); never weight across a config boundary.
- Add mocked-provider tests: equal vs weighted on a unanimous set (no change), on a contested set with one correct independent minority vs a correlated wrong bloc (weighted recovers the minority), and a config-change cache-invalidation test.


## Treat high council agreement as a possible-monoculture warning, not automatic high confidence [2606.00005]

**Source**: [[emergent-collaborative-deliberation-in-multi-model-ai-systems]] (paper note) — arXiv:2606.00005, "Emergent Collaborative Deliberation in Multi-Model AI Systems: A BFT-Derived Protocol for Human-Moderated Epistemic Synthesis" (VD Doske / Consilia), https://arxiv.org/pdf/2606.00005
**Implementation Layer**: `backend/council.py` (`compute_council_confidence`, `stage3_synthesize_final` hedge/attribution instruction, deep-mode `stage2a_collect_critiques`), `backend/config.py` thresholds, and `tests/test_council_confidence.py`.
**Missing Capability**: `compute_council_confidence` treats high top-1 stability / strong rank agreement as **high confidence**, and the chairman only hedges on *disagreement*. There is no countermeasure for the opposite failure: unanimous agreement that reflects shared training-data priors (RLHF "preference collapse") rather than independent verification — i.e., a monoculture confidently agreeing on a contested or subjective claim.
**Architecture Evidence**: `CLAUDE.md` documents `compute_council_confidence` (top-1 stability, Kendall-like agreement, disagreement) feeding the chairman's "warn when split" instruction and auto-standard escalation — all of which interpret agreement as good. The repo already structurally fights correlated models (chairman/council family heterogeneity startup check), which is the same monoculture concern this paper raises at the *output-agreement* level rather than the *family* level.
**Benefit Hypothesis**: When the council is unanimous AND the question is contested/subjective (opinion, policy, forecasting, value-laden), the chairman is instructed to add an inverse hedge — "the council agreed, but on this kind of question agreement may reflect shared training priors rather than independent confirmation" — then on a labelled probe of contested prompts the rate of overconfident unanimous-but-wrong syntheses drops, with no added hedging on factual/verifiable unanimous answers. No latency cost (reuses existing Stage 2 signals + a cheap claim-type check already present for mode selection).
**Confidence**: 0.4
**Reasoning**: The paper's headline reframing — "high convergence index is a warning signal, not a success metric; narrow spread indicates either genuine consensus OR shared bias masquerading as agreement" — is directly relevant and is backed by well-established cited literature (Kirk et al. RLHF diversity collapse; Xie et al. preference collapse; Santurkar et al. >99% dominant-opinion reflection; Wynn et al. multi-agent debate can *decrease* accuracy via conformity). That cited basis is solid independent of this paper's own rigor. Honest caveats lowering confidence: the paper itself is a single-author ("Consilia") preprint with self-reported empirical batteries (1,478 sessions, cost figures, a bespoke "convergence index") that I could not independently verify, and its grand epistemic framing exceeds what a QA council needs. The actionable nugget is narrow: an inverse hedge on contested unanimous runs, plus optionally an explicit adversarial-challenger disposition in deep-mode critique to structurally guarantee challenge (countering sycophancy/disagreement-collapse).
**Why Not Already Tried**: Every confidence/escalation item in this plan equates agreement with confidence; none models the case where agreement is the failure. The chairman hedge is one-directional (disagreement only).
**Supporting Evidence (folded in)**: Two further notes corroborate that council agreement can be false consensus. (1) *Argument Collapse* (arXiv:2606.01736) measures that frontier LLMs answering the same public-debate prompt converge hard: only 3.4% of LLM main arguments are unique within a debate vs 65.3% for humans (sub-arguments 9.1% vs 41.0%), and even explicit diversity prompting recovers only ~50–55% of the human argument space — so multiple council members can produce *apparent* diversity while collapsing to the same few arguments. (2) *The Inverse-Wisdom Law* (arXiv:2604.27274) argues agentic swarms can prioritize architectural agreement over truth ("Consensus Paradox", "Heterogeneity Mandate at the synthesizer node") — directionally aligned with the repo's chairman-heterogeneity invariant, though that paper is a grandiose, low-rigor preprint (sweeping "laws", fictional model versions), cited only as motivation, not evidence.
**Pairs With**: [2510.01499] correlation-aware aggregation (same monoculture concern, aggregation side), [2606.03189a] dual-order debiasing, [2606.03137] (spiral-of-silence/conformity anti-pattern), and the chairman-heterogeneity invariant (`tests/test_chairman_heterogeneity.py`).

### Proposed Changes

- Add a contested/subjective claim-type signal (reuse the existing auto-mode claim classifier) and combine it with high `compute_council_confidence` to detect "unanimous-on-contested" runs.
- Extend the chairman synthesis prompt with an inverse-hedge branch for unanimous-on-contested runs, distinct from the existing low-confidence (split) hedge; keep factual/verifiable unanimous answers unhedged.
- Optionally (deep mode): assign at least one critic in `stage2a_collect_critiques` an explicit adversarial-challenger disposition so challenge is structurally guaranteed regardless of which models seat — a BFT-style "≥1 adversarial" composition rule; measure whether it reduces revision conformity (ties into the Stage 2b conformity-metric idea).
- Add mocked-provider tests: unanimous-factual (no inverse hedge), unanimous-contested (inverse hedge fires), split-contested (existing hedge), and a check that the inverse hedge never suppresses a correct verifiable claim.
- (From Argument Collapse, arXiv:2606.01736) Add an optional Stage 1 argument-diversity probe: measure inter-member overlap of distinct claims/arguments; when members agree on the *ranking* but their Stage 1 content has collapsed to near-identical arguments, treat that as additional evidence of monoculture rather than independent corroboration, and feed it into the inverse-hedge decision.


## Intrinsic-consistency reliability gate for the judge before trusting its scores [2602.00521]

**Source**: [[diagnosing-the-reliability-of-llm-as-a-judge-via-item-response-theory]] (paper note) — arXiv:2602.00521, "Diagnosing the Reliability of LLM-as-a-Judge via Item Response Theory" (Choi, Park, Cho, Park, Kim; Chung-Ang / Seoul National Univ), https://arxiv.org/pdf/2602.00521
**Implementation Layer**: `backend/eval/judge.py` (`judge.v1` schema, judge invocation, `available` flag), `scripts/judge_eval_smoke.py`, `tests/test_judge_eval.py`.
**Missing Capability**: The judge runs one rubric prompt and reports scores, with `available=false` only on hard failure (parse/timeout). It never checks whether the judge is an *internally consistent measurement instrument* — i.e., whether its scores stay stable under semantically-equivalent prompt perturbations. A judge whose verdict flips on a trivial reword is unreliable, but today its scores are reported as clean measurement regardless.
**Architecture Evidence**: `CLAUDE.md` frames `backend.eval.judge` as a measurement artifact (not part of the answer contract) that validates every rubric score under `judge.v1` and returns `available=false` on failure. That `available` flag is the natural carrier for a reliability verdict, but it currently encodes only "did the call succeed," not "is this measurement trustworthy."
**Benefit Hypothesis**: When the judge optionally re-scores under a few semantically-equivalent prompt perturbations (paraphrase, rubric-item reorder, whitespace/typo noise) and computes a coefficient-of-variation–style stability proxy, then runs whose judge is prompt-unstable are flagged `reliability="low"` (or `available=false`) instead of silently reported, and on a labelled probe the judge verdicts that survive the gate agree with reference judgments more often than the ungated set. The paper's sequential design — interpret alignment only after intrinsic consistency holds — becomes an explicit gate rather than an unstated assumption.
**Confidence**: 0.45
**Reasoning**: The paper grounds judge reliability in two psychometric phases — intrinsic consistency (prompt-consistency `C_V` < 0.1; marginal reliability ρ ≥ 0.7) then human alignment (θ_ratio, Wasserstein `D_W`) — and shows empirically that "there is no free lunch": across G-Eval/HelpSteer-2/etc. no model passes consistency on all criteria, detailed-instruction + CoT prompts reduce `C_V`, and judges systematically perceive quality differences *more broadly* than humans (θ_ratio > 1, i.e. they amplify margins). Two portable lessons: (a) verify a judge is stable under perturbation before trusting its score, and (b) don't over-read judge score *deltas* as true quality gaps. Honest bound on confidence: the full method (fitting a Graded Response Model via NUTS/PyMC, many samples per evaluation subject) is far too heavy for the council's per-synthesis operator judge; only the lightweight perturbation-`C_V` gate and the detailed-prompt/CoT guidance are practical here. The gate also costs extra judge calls (mitigated: judge is off the `ask_council` hot path).
**Why Not Already Tried**: The judge item [2604.011] established a single-rubric measurement; no plan item asks whether the judge itself is a reliable instrument. The `available` flag conflates call-success with measurement-trust.
**Pairs With**: [2606.03189b] rubric-ensemble (its rubric variants ARE perturbations — compute their `C_V`/spread as the reliability signal, not just a majority vote), [2606.03189a] dual-order debiasing (order is one perturbation axis), and [2604.011] judge harness.
**Related (folded in)**: FACT-E (arXiv:2604.10693) argues LLM self-evaluation of reasoning is confounded by internal bias and uses instrumental-variable–like *controlled perturbations* of a chain to separate genuine step-to-step dependency from bias-driven artifacts, scoring intra-chain faithfulness + CoT-to-answer consistency. ### Proposed Changes

- Add an optional reliability pass to `backend/eval/judge.py`: re-score the same (candidate, baseline) under K semantically-equivalent prompt perturbations and compute a stability proxy (coefficient of variation of the score), without fitting full Bayesian IRT.
- Extend the judge result with a `reliability` field (`high`/`low`) derived from the stability proxy; when unstable, surface `reliability="low"` so downstream consumers (cache/routing quality gates from [2604.011]) discount the verdict instead of trusting it.
- Adopt the paper's prompt guidance for the default judge rubric: detailed criteria + chain-of-thought reduce `C_V`; prefer a 5-point over a 7-point scale where a scale is used.
- Document the θ_ratio>1 caveat near the judge: LLM judges amplify quality gaps relative to humans, so treat judge score *margins* as ordinal signal, not calibrated magnitude.
- Reuse the [2606.03189b] rubric variants as the perturbation set so the ensemble and the reliability gate share one fan-out; add `tests/test_judge_eval.py` cases for stable-judge (passes), unstable-judge (flagged), and all-fail (available=false).

**Source**: [[selfdoubt-uncertainty-quantification-for-reasoning-llms-via-the-hedge-to-verify-ratio]] (paper note) — arXiv:2604.06389, "SELFDOUBT: Uncertainty Quantification for Reasoning LLMs via the Hedge-to-Verify Ratio", https://arxiv.org/abs/2604.06389
**Implementation Layer**: `backend/council.py` (`compute_council_confidence`, Stage 1 response post-processing, chairman down-weighting, escalation/routing predicates), an offline per-model marker-calibration step over stored responses in `backend/storage.py`.
**Missing Capability**: `compute_council_confidence` is built only from cross-member *ranking agreement*; it has no per-member *self-uncertainty* signal. **Pairs With**: [2510.01499] (per-member reliability weights — HVR is a per-response complement to per-model accuracy), [2606.02981]-style escalation gating, and quick-mode self-check (the no-hedge gate could validate a quick-mode answer).

### Proposed Changes

- Add an offline calibration step that mines per-model hedge/verify marker dictionaries from stored Stage 1 responses (multi-LLM seed + n-gram expansion as in the paper); persist them keyed by model id.
- Compute a per-response HVR + verbalized-confidence score during Stage 1 post-processing (no extra model calls) and store it in response metadata.
- Fold the per-member signal into `compute_council_confidence` as an additional input and expose it to the chairman so heavily-hedged members are down-weighted; treat a unanimous no-hedge as a confidence booster.
- Gate use behind config and validate against judge scores before trusting it; add tests for hedge-heavy, no-hedge, and trace-absent responses.

**Source**: [[raser-recoverability-aware-selective-escalation-router-for-multi-hop-question-answering]] (paper note) — arXiv:2606.02488, "RASER: Recoverability-Aware Selective Escalation Router for Multi-Hop Question Answering", https://arxiv.org/pdf/2606.02488
**Implementation Layer**: `backend/agent_router.py`, the auto-mode confidence-escalation predicate in `backend/council.py`, `backend/metrics.py`, `scripts/agent_routing_benchmark.py`.
**Missing Capability**: Escalation (auto-standard → deep) and sparse→full council expansion are driven by deterministic thresholds plus a capped `TITLE_MODEL` classifier. **Why Not Already Tried**: Existing routing/escalation items use hand-tuned thresholds and a capped classifier; none learns a router from outcome data or models recoverability to avoid spending compute on questions that cannot be helped. **Pairs With**: the confidence-escalation item, adaptive routing in `agent_router.py`, "Compute-aware early stopping", and [2606.02866] (its deep-benefit predicate is the same "is more compute worth it" question).
**Related (folded in)**: Two papers point at the same routing signal from a different angle but require heavier machinery than a GBM. *Differentiable Mixture-of-Agents* (arXiv:2605.15706) does step-wise sparse agent activation supervised by **predictive entropy** — a candidate routing feature worth adding, though its router is trained end-to-end (heavier than our frozen-model GBM). *Squeeze-Evolve* (arXiv:2604.07725) routes evolutionary operators to cost-effective models via confidence-derived fitness, reserving expensive models for high-marginal-utility steps (up to ~3× cheaper / ~10× throughput) and flags **diversity collapse** as the bottleneck — i.e. reserve the expensive chairman for high-utility synthesis and route cheaper stages elsewhere, a cost-aware extension of this router that also ties back to the monoculture concern [2606.00005]. Two more reinforce the *learned* over *hand-tuned* threshold: *Act or Escalate?* (arXiv:2604.08588) finds escalation thresholds are model-specific and miscalibrated in model/domain-specific ways, not predicted by scale, and that prompt cost-framing alone is weak while CoT-target training generalizes — directly arguing the council's single `COUNCIL_LOW_CONFIDENCE_TOP1_THRESHOLD` is likely uncalibrated and a learned router should beat it. *Agent Capsules* (arXiv:2605.00410) shows granularity reduction (merging/sparsifying agents) silently degrades quality via tool loss and prompt compression unless gated by an explicit quality check — a direct caution for the shipped sparse-routing path (commit eadbeee): sparse→full expansion needs a quality gate, exactly what this router provides.

### Proposed Changes

- Define a cheap feature vector per run (Stage 1 success count, inter-member agreement, response length/hedging stats, `compute_council_confidence` fields) computable without extra model calls.
- Train an offline GBM on stored runs labeled by judge outcome to predict whether escalation/expansion improves quality; ship it behind config with the deterministic threshold path as fallback.
- Add a recoverability check: when the router predicts no gain (and historically similar runs did not improve under deep mode), skip escalation and record the reason in metrics/metadata.
- Extend `scripts/agent_routing_benchmark.py` to report quality-preserved vs calls-saved for router vs deterministic vs always-escalate, and log dropped/declined escalations (no silent caps).


## MBR consensus-centroid selection with risk-aware abstention as a cheap Stage 1 signal [2604.11141]

**Source**: [[reducing-hallucination-in-enterprise-ai-workflows-via-hybrid-utility-minimum-bayes-risk-humbr]] (paper note) — arXiv:2604.11141, "Reducing Hallucination in Enterprise AI Workflows via Hybrid Utility Minimum Bayes Risk (HUMBR)", https://arxiv.org/pdf/2604.11141

- Reuse EGS as the evidence-grounding component of the [2509.26184] attribution-support check (shared "is the quoted span real" logic), and trial the same SVA/hedging signals on council-member Stage 1 traces feeding `compute_council_confidence`.
