# Plan to Improve LLM Council

Based on the paper "Council Mode: Collaborative LLM Agents for Multi-Model Deliberation" (2604.02923v1).

## 1. Triage-классификатор (приоритет: высокий)

**Что:** Лёгкая модель (`TITLE_MODEL`) классифицирует входящий запрос перед запуском полного council. Простые запросы (приветствия, однозначные факты, короткие вопросы) обрабатываются напрямую одной моделью, минуя Stage 1-3.

**Зачем:** Статья показывает снижение средней латентности ~30% без потери качества (Table 4). Сейчас каждый запрос проходит полный пайплайн — это расточительно для тривиальных вопросов.

**Где менять:**
- `backend/council.py` — новая функция `triage_query()` перед `run_full_council()`
- `backend/main.py` — вызов triage перед запуском council
- Frontend — индикация, что запрос обработан в упрощённом режиме

**Примерная логика:**
```
TITLE_MODEL классифицирует запрос → {simple, moderate, complex}
- simple → ответ от одной модели (CHAIRMAN_MODEL)
- moderate → стандартный 3-stage pipeline
- complex → thorough mode (5 stages)
```

## 2. Структурированный синтез Chairman (приоритет: высокий)

**Что:** Заменить свободный промпт chairman на принудительную декомпозицию ответа с 4 секциями.

**Зачем:** Ablation из статьи: замена структурированного синтеза на majority vote увеличивает галлюцинации на 32.7%. Явная структура заставляет chairman анализировать расхождения между моделями, а не просто выбирать "лучший" ответ.

**Где менять:**
- `backend/council.py` → `stage3_synthesize_final()` — новый промпт для chairman

**Новая структура промпта:**
```
1. Consensus — факты/выводы, по которым все эксперты согласны
2. Disagreements — где эксперты противоречат друг другу и почему
3. Unique Findings — инсайты, которые дала только одна модель
4. Final Analysis — итоговый ответ с учётом всего вышеперечисленного
```

**Нюанс:** Секции 1-3 нужны для внутренней "рефлексии" chairman, но пользователю показывать только Final Analysis (или дать опцию раскрыть полный breakdown в UI).

---

Inspired by "Nidus: Externalized Reasoning for AI-Assisted Engineering" (2604.05080v1).

> Nidus — система формальной верификации для AI-assisted разработки. Основная идея: инженерные инварианты нельзя надёжно поддерживать как learned behavior — их нужно enforce извне через decidable проверки (SMT-солвер Z3). Каждая мутация артефакта проверяется до персистирования. Прямого отношения к LLM deliberation нет, но две идеи переносимы.

## 3. Friction Ledger — трекинг качества моделей (приоритет: средний)

**Что:** Заимствование концепции friction ledger из Nidus. Вести лог качества каждой модели на основе peer-ранжирования: как часто модель оказывается на первом месте, как часто на последнем, как часто её ответ расходится с консенсусом.

**Зачем:** Nidus использует friction ledger для динамического вычисления trust tier агентов — агенты с высоким friction теряют доступ к сложным задачам. Аналогично, в council можно использовать накопленную статистику для:
- Взвешенного агрегирования ранжирований (голос стабильно сильной модели весит больше)
- Адаптивного выбора chairman (автоматически назначать модель с лучшим track record)
- Отображения в UI (пользователь видит, какие модели надёжны по историческим данным)

**Где менять:**
- `backend/storage.py` — новая таблица `model_performance` (model, avg_rank, total_queries, first_place_count, last_place_count)
- `backend/council.py` → `calculate_aggregate_rankings()` — записывать результаты в таблицу после каждой делиберации
- `backend/main.py` — новый эндпоинт `GET /api/models/stats`
- Frontend — страница или виджет с аналитикой моделей

**Примерная схема:**
```sql
CREATE TABLE model_performance (
    model TEXT,
    query_date TEXT,
    rank_position REAL,
    was_first INTEGER,
    was_last INTEGER
);
-- Агрегация: SELECT model, AVG(rank_position), COUNT(*) FROM model_performance GROUP BY model
```

## 4. Верификация пайплайна — proof obligations для council (приоритет: низкий)

**Что:** Заимствование идеи decidable verification gates. Добавить формальные проверки на каждом этапе пайплайна council, которые гарантируют корректность до того, как результат передаётся дальше.

**Зачем:** Сейчас council продолжает работу даже если Stage 2 вернул непарсируемые ранжирования. Nidus показывает, что verification before persistence (проверка до персистирования) исключает наблюдение невалидного состояния другими участниками системы.

**Конкретные проверки:**
- Stage 1 → Stage 2: минимум N успешных ответов (сейчас есть, но порог = 1)
- Stage 2 → Stage 3: все parsed_ranking непустые и содержат все Response labels
- Stage 2 → aggregate: ранжирования консистентны (нет дубликатов, все модели учтены)
- Thorough mode: Stage 2a critiques содержат секции для каждого Response

**Где менять:**
- `backend/council.py` — новая функция `verify_stage_output()` между этапами
- При провале проверки — повторный запрос к failed модели (retry) или исключение из агрегации с логированием
